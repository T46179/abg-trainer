Executive Summary

The highest-risk issue is that the app ships the full answer bank to every browser. Each case includes `answer_key`, diagnosis, compensation data, and explanation, and the frontend preloads the entire dataset at startup. For an ABG trainer whose core value is stepwise interpretation, that makes answer extraction trivial.

The second major issue is state trust. XP, level, streak, badges, recent results, and daily usage are loaded from and saved back to unsigned `localStorage`, with no validation beyond a few shape checks. A moderately technical user can self-award progression, fake stats, and bypass daily limits.

The third major issue is reward duplication. `finishCase()` awards XP entirely client-side and is not idempotent. In a classic non-module script, that makes infinite XP farming from the same completed case realistically possible through DevTools.

The fourth issue is entitlement/control logic. Difficulty access, Learn access, daily case limits, and testing-mode behavior are all enforced in frontend JS from mutable runtime state and generated JSON. In this current build, `TESTING_MODE` is already enabled and the UI explicitly says all difficulties are unlocked with 5x XP.

The fifth issue is telemetry leakage. GA/Clarity are enabled, and the app sends `case_id`, `archetype`, step correctness, timing, and feedback-open events. That leaks internal content/state to third parties and gives technical users an extra side channel for the current case.


Detailed Findings Table

1. Title: Full answer bank shipped to client
Severity: Critical
Category: data exposure, anti-cheat
Exact files / functions involved:
- `generator/generators/common.py`: `build_answer_key`, `build_case`
- `docs/app.js`: `isCorrectAnswer`, `loadCases`, app init
How the exploit works:
1. The generator includes `answer_key`, `expected_compensation`, and explanation inside every case object.
2. The frontend loads the full `abg_cases.json` payload at startup.
3. A user opens DevTools, fetches the JSON, or inspects runtime state.
4. The user reads the correct answer fields before or during the case.
Realistic impact:
- Perfect scores on demand
- Full answer-bank scraping
- No secure exam mode
- Automated solver scripts are straightforward
Recommended fix:
- Serve cases from a backend and keep scoring/answers server-side for any scored or protected mode.

2. Title: XP, level, streak, stats, and badges trust unsigned localStorage
Severity: High
Category: anti-cheat, logic flaw
Exact files / functions involved:
- `docs/app.js`: storage keys, `loadUserState`, `saveUserState`, init
How the exploit works:
1. Open DevTools Application/Storage.
2. Edit `abgmaster_userState`.
3. Set arbitrary `xp`, `level`, `casesCompleted`, `streak`, `dailyCasesUsed`, `recentResults`, or badges.
4. Reload the page.
5. The app merges the forged object and persists it again.
Realistic impact:
- Fake progression
- Fake profile stats
- Reset or bypass daily limits
- Fake leaderboard position
Recommended fix:
- Move progression and achievement authority to a backend; at minimum stop treating local state as authoritative.

3. Title: Infinite XP by re-calling `finishCase()`
Severity: Critical
Category: anti-cheat, replay
Exact files / functions involved:
- `docs/index.html`: classic script load
- `docs/app.js`: `finishCase`
How the exploit works:
1. Complete any case once.
2. Open DevTools.
3. Re-invoke `finishCase()` from the global scope.
4. Each call re-awards XP, increments counters, updates streak/daily usage, and saves state.
5. Repeat indefinitely.
Realistic impact:
- Infinite XP from one case
- Artificial level progression
- Polluted analytics
Recommended fix:
- Make completion idempotent and tie rewards to a one-time server-issued attempt ID.

Attack Scenarios

1. Granting yourself XP via console/localStorage
1. Open the site and complete one case normally.
2. Open DevTools and inspect `localStorage.abgmaster_userState`.
3. Set `xp` to a large number or directly re-run `finishCase()`.
4. Reload or repeat.
5. The app recalculates level from forged XP and shows the new progression.

2. Unlocking Master difficulty instantly
1. Open DevTools.
2. Edit `abgmaster_userState` and set `xp` high enough to reach level 20, or directly set a high `level`.
3. Reload.
4. The frontend recalculates unlocked difficulties from local state.
5. Master appears unlocked.
Note: in the current build this is even easier because testing mode already unlocks all difficulties.

3. Replaying cases for infinite rewards
1. Complete a case once and observe the correct answers.
2. Call `finishCase()` again from DevTools, or keep repeating known cases since rewards are not tied to a unique attempt.
3. Each replay adds XP and counters again.
4. Repeat until the desired level is reached.

4. Extracting answers from loaded data
1. Open DevTools Network or Application/Memory tools.
2. Inspect `abg_cases.json`.
3. Read `answer_key`, `final_diagnosis`, `expected_compensation`, and explanation for each case.
4. Use that information during the session or export the whole answer bank.

5. Keeping TESTING_MODE advantages
1. Load the app.
2. The JSON payload contains `testing_mode: true` and `testing_xp_multiplier: 5`.
3. The frontend accepts these values and bypasses normal restrictions.
4. All difficulties and 5x XP remain active.

6. Bypassing progression/difficulty systems
1. Open DevTools.
2. Patch local state or runtime objects so the app thinks you are premium/high-level.
3. Use the difficulty selector or call `startNewCase("master")`.
4. Because access control is frontend-only, the UI accepts the manipulated state.


Trust Boundary Assessment

What the frontend currently trusts that is unsafe
- It trusts `abg_cases.json` for secret answer content.
- It trusts `localStorage` for XP, level, streak, daily counts, badges, and stats.
- It trusts the client clock for streaks and daily quotas.
- It trusts frontend JS for difficulty, Learn access, and testing-mode entitlement behavior.

What cannot be secured in a static client-only app
- Hidden answers
- Secure scoring
- Paywalls and difficulty locks
- One-reward-per-attempt guarantees
- Daily quotas and streak integrity
- Real leaderboard integrity
- True anti-cheat enforcement

What is partially mitigatable vs fundamentally insecure
- Partially mitigatable:
  - Reducing telemetry leakage
  - Removing obvious diagnosis-bearing identifiers
  - Removing hidden reward paths
  - Adding local validation/clamping
  - Making casual cheating less convenient
- Fundamentally insecure without backend authority:
  - Answer secrecy
  - Progression integrity
  - Entitlement enforcement
  - Replay protection
  - Competitive ranking


Hardening Strategy

Immediate quick wins (conceptual only)
- Stop shipping public builds with testing mode enabled.
- Remove diagnosis-bearing IDs and per-step correctness from analytics.
- Remove or fully disable hidden timed-mode reward behavior.
- Stop presenting leaderboard/profile stats as authoritative.
- Clearly frame the current static build as practice-only if secure scoring is not available.

Medium-term improvements
- Separate practice mode from scored mode.
- Stop preloading the full case bank at startup.
- Issue cases per attempt rather than as one static downloadable payload.
- Add one-reward-per-attempt logic and solved-case tracking.
- Reduce answer and explanation exposure until after a scored attempt is finalized.

Backend-required protections
- Server-side case delivery
- Server-side answer checking and scoring
- Server-owned progression and entitlement state
- Server time for streaks and daily limits
- Server-issued attempt/session IDs
- Server-side leaderboard records


Priority Fix List

1. Stop shipping `answer_key` and explanations to the client for scored sessions.
2. Make case completion idempotent and impossible to replay for reward.
3. Turn off public `TESTING_MODE` immediately.
4. Move XP/level/streak/unlocks off `localStorage`.
5. Stop sending `case_id`/`archetype`/step correctness to GA and Clarity.
6. Add a server-issued attempt ID and one-reward-per-attempt enforcement.
7. Replace client-clock streak/daily logic with server time.
8. Remove hidden timed-mode reward logic.
9. Stop using client-generated/mock leaderboard data as if it were meaningful.
10. Split practice feedback mode from any mode that claims progression integrity.


What cannot be secured without a backend

You cannot securely hide answers, enforce XP/level progression, prevent replay rewards, enforce paywalls or difficulty locks, trust streak/daily limits, or run a real leaderboard in a static client-only app. You can only make cheating harder.

For this codebase, “harder to cheat” means obfuscation and UI friction. “Actually secure” requires a backend that owns answers, attempts, entitlements, time, and rewards.
